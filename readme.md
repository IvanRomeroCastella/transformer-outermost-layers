# What Outermost Layers Do

A cross-architecture mechanistic study of trained transformers.

This repository contains the code that reproduces the figures and numerical
results of the paper *"What Outermost Layers Do: A Cross-Architecture
Mechanistic Study of Trained Transformers"* (arXiv preprint, 2026).

## What this repo is about

Trained transformers — DistilBERT, BERT, and GPT-2 — show a consistent
**sandwich pattern** across architectures: the first and last layers behave
qualitatively differently from the central plateau in between. This repo
characterizes that pattern and then dissects what the two outer layers (the
*translators*) actually do mechanistically.

Three findings shape the paper, and each one falsified an initial hypothesis
in the opposite direction:

- **The sandwich is general, not architectural folklore.** It appears in
  all three models, and **gets more extreme**, not more distributed, in
  deeper architectures.
- **Translators are geometric dispersers, not concentrators.** The central
  plateau concentrates discriminative information along a small number of
  high-variance directions. The translators redistribute that information
  across a wider high-dimensional manifold misaligned with the natural
  variance directions. Recovering POS or lexical identity from the exit
  layer requires substantially **more** principal components than recovering
  it from the core.
- **GPT-2's exit translator is doing something the linguistic probes can't
  see.** At the final layer, PC1 captures roughly 35% of variance along a
  direction that is orthogonal to POS, frequency, position, and sentiment.
  That dominant direction is mechanistically tied to next-token preparation
  — a structurally novel finding relative to encoder behavior.

The repo is organized into two self-contained blocks. The first
(`01_sandwich_geometry/`) measures the sandwich pattern globally across the
three models. The second (`02_translator_mechanism/`) characterizes what the
two outer layers do, using probing (A1), dimensional budget (A2), prober
weight analysis (A3), and ablation sensitivity (B1).

Pre-registered hypotheses and their outcomes for each block are documented
in `01_sandwich_geometry/prereg.md` and `02_translator_mechanism/prereg.md`.

```
│   ├── metrics_m3.py               M3
│   └── visualize.py
├── 02_translator_mechanism/
│   ├── README.md                   block-specific instructions
│   ├── prereg.md                   hypotheses and outcomes
│   ├── common.py                   shared utilities for this block
│   ├── b1_ablation.py              B1: ablation sensitivity
│   ├── extract_probing_data.py     prepares UD-EWT and SST-2 activations
│   ├── a1_probing.py               A1: classical layer-wise probing
│   ├── a2_budget.py                A2: dimensional-budget probing
│   ├── a3_weights.py               A3: prober-weight PC1 alignment
│   ├── fix_freq_bucket.py          post-hoc fix for freq_bucket labels
│   └── analyze.py                  figures and numerical summary
```

Figures referenced by the paper are produced by `01_sandwich_geometry/
visualize.py` (Figure 1, written to `01_sandwich_geometry/results/figures/`)
and `02_translator_mechanism/analyze.py` (Figure 2 and supporting panels,
written to `02_translator_mechanism/results/figures/`). They are
regenerated by the pipeline and are not tracked in the repo.

`02_translator_mechanism/` depends on the embeddings and datasets produced
by `01_sandwich_geometry/` — specifically, it reads
`01_sandwich_geometry/data/{model}/dataset_*.jsonl` and
`embeddings_*.npz`. Run block 1 before block 2.


## Models and data

The three pretrained models are fetched from HuggingFace using the
following specific commits, which we recommend pinning for exact
reproducibility:

| Model | HuggingFace name | Commit |
|---|---|---|
| DistilBERT | `distilbert-base-uncased` | `12040accade4e8a0f71eabdb258fecc2e7e948be` |
| BERT       | `bert-base-uncased`       | `86b5e0934494bd15c9632b12f734a8a67f723594` |
| GPT-2      | `gpt2`                    | `607a30d783dfa663caf39e06633721c8d4cfcd7e` |

Datasets are fetched at runtime:

- **Block 1** uses a controlled Wikipedia-derived dataset constructed by
  `build_datasets.py` (coherent / permuted / random conditions, 500
  sentences each, 20 content tokens per sentence).
- **Block 2** additionally uses UD English-EWT (downloaded directly from
  the official `UniversalDependencies/UD_English-EWT` GitHub repo as
  CoNLL-U) and SST-2 (loaded via `datasets`).

This repo does not ship data files. Every `.npz` and `.jsonl` consumed by
the analysis is generated on your machine by running the pipeline.

## Hardware and software

The pipeline was developed and tested on:

- **GPU**: NVIDIA RTX 5060 Ti (16 GB VRAM)
- **CPU / RAM**: Intel i7-14700F, 64 GB RAM
- **OS**: Ubuntu 24.04
- **Python**: 3.13
- **PyTorch**: 2.x with CUDA

End-to-end runtime on this configuration: roughly 30–60 minutes for the
full pipeline (both blocks). The bottleneck is the ablation sweep in B1
(~10–20 min) and the per-layer probing in A1/A2/A3 (~10–15 min combined).

It should run on any CUDA GPU with at least 8 GB of VRAM; smaller GPUs
will work but you may need to reduce `BATCH_SIZE` in
`02_translator_mechanism/b1_ablation.py` and `a1_probing.py`. CPU-only
execution is technically possible but impractical for the ablation sweep.

## Quick start

```bash
# Clone and install dependencies.
git clone https://github.com/IvanRomeroCastella/transformer-outermost-layers.git
cd transformer-outermost-layers
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Run the full pipeline (both blocks, ~30-60 min on RTX 5060 Ti).
python run_pipeline.py

# Open pipeline_report.html in a browser to see the consolidated report.
# Raw data in pipeline_results_raw.txt
```

`run_pipeline.py` runs the 13 pipeline scripts in dependency order, verifies
inputs and outputs at every step, captures timings and errors, and writes a
single self-contained HTML report (`pipeline_report.html`) with the status
table, all generated figures embedded inline, and the numerical summaries
from `02_translator_mechanism/results/summary.json` formatted as readable
tables. Re-running it skips steps whose outputs already exist; pass
`--rerun` to force regeneration of everything, or `--dry-run` to inspect
what would run without executing anything.

If you prefer to run scripts manually, each block has its own `README.md`
with the step-by-step commands.

## Reproducibility notes

- Random seeds are set in scripts that involve training (probers in
  `a1_probing.py` use `torch.manual_seed(42)`). PCA and SVD operations are
  deterministic on a given device.
- Numerical results are stable across runs to within float32 precision
  (relative tolerance ~1e-4) on the tested hardware. Cross-platform results
  may vary slightly due to CUDA library versions and BLAS implementations.
- `fix_freq_bucket.py` exists because the initial bucketization in
  `extract_probing_data.py` bucketizes by word rank rather than token
  count, which produces an unbalanced majority baseline for the
  `freq_bucket` label. The fix re-bucketizes by token-count percentiles
  so each bucket holds roughly one third of the tokens. Run the fix once
  after extraction and before the probing scripts.
